测试计划

XXX 项目测试计划

1. 引言

1.1 编写目的

根据需求文档，提炼测试功能点、制定测试策略、评估测试风险，预估编写测试用例、执行功能测试和回归测试的工作量，进行人员和进度安排。

1.2 预期读者

管鑫所项目经理、开发、产品、测试。

1.3 参考资料

需求文档

2. 测试范围

3. 测试策略

3.1 UI 测试

3.1.1 文案测试

（1）字体
（2）字号
（3）格式
（4）规范

3.1.2 图片测试

（1）清晰
（2）尺寸
（3）配色
（4）风格

3.1.3 布局测试

（1）尺寸大小
（2）位置合理
（3）排序规律
（4）对齐方式

3.1.4 控件测试

（1）输入框
（2）按钮
（3）下拉菜单
（4）列表
(5) 单选框
（6）复选框
（7）时间框
（8）左右选择框
（9）树形
（10）图标类检查
（11）文件类
（12）图片类
（13）URL 检查
（14）IP 输入框

3.1.5 快捷键测试

3.2 功能测试策略

具体见每个版本测试用例。

3.3 系统兼容性测试

3.4 易用性测试

3.5 安全性测试

3.6 网络环境测试

网络的影响会影响到用户的体验，一般遵守 258 原则是最好的。

3.7 单元测试

由开发人员完成。
3.3 性能测试
关键接口功能，简单做性能测试。

4. 测试资源

4.1 测试人员

4.2 测试环境

4.3 测试工具

4.3.1 Bug 管理工具：bugfree

4.3.2 postman

前期测试呼叫系统，使用 postman 模拟相应的用户场景进行测试。

4.3.3 IETester

测试 IE 浏览器的不同版本。

4.3.4 Vbox 虚拟机

4.3.5 Jemter

重要端口、步骤、流程做压力测试。

5. 进度安排

5.1 测试进度及工作量估算

根据周会原型确定周任务：
例如
任务 时间 执行人员 预计工作量 / 天
编写测试计划 2019-07-10 张三 1 天
编写测试用例 2019-07

根据具体的需求、开发计划估算具体的测试工作量。

5.2 输出文档

（1）测试计划
（2）测试用例（bugfree 中查看）
（3）测试数据（上传至 bugfree）
（4）测试报告

6. 发布标准

6.1 测试完成标准

（1）完成所有测试类型。

（2）没有影响用户正常使用的 Bug。

（3）bug 少于一定的数量（需要商定）

（4）服务端性能满足设计目标。

6.2 产品发布标准

（1）所有产品需求都已完成。

（2）遗留 bug 经过风险评估。

（3）有完备的流程文档。
